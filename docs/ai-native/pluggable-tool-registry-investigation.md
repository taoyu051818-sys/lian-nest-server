# Pluggable Tool Registry Investigation

Investigates whether LIAN's self-cycle runner should adopt a YAML-based
tool registry pattern inspired by SWE-agent, where tools are declared as
config with structured input/output schemas and loaded at startup —
instead of the current hardcoded PowerShell orchestration.

> **Closes:** [#1495](https://github.com/taoyu051818-sys/lian-nest-server/issues/1495)

## External Pattern: SWE-agent YAML Tool Registry

SWE-agent defines tools via YAML config. Each tool bundle declares:

- **Identity** — name, description
- **CLI** — executable path and invocation pattern
- **Input schema** — structured parameters with types and constraints
- **Output schema** — expected return shape
- **Registry** — tools are first-class citizens discovered at startup

Adding a new tool means adding a YAML file. The agent loop is
config-driven and never edited for tool changes.

### Key Properties

| Property | SWE-agent Approach |
|----------|-------------------|
| Tool definition | YAML file per tool |
| Schema | Structured input/output with types |
| Discovery | Registry loaded at startup |
| Loop modification | None — config-driven |
| Tool identity | Stable name in YAML |
| Blast radius | Isolated per YAML file |

## Current LIAN State

### Self-Cycle Runner Architecture

The self-cycle runner (`run-self-cycle.ps1`) orchestrates five scripts
in sequence:

| Step | Script | Role |
|------|--------|------|
| 1 | `state-reconciler.ps1` | Drift detection |
| 2 | `write-main-health-state.ps1` | Health marker (pre-cycle) |
| 3 | `check-launch-gate.ps1` | Task validation |
| 4 | `batch-launch.ps1` | Worker dispatch |
| 5 | (built-in) | Summary |

Each script is a standalone PowerShell or Node.js tool with its own CLI,
parameters, and output format. The runner calls them in order and
surfaces human decision stop points between steps.

### Control Skill Registry (Governance Layer)

LIAN already has a governance-layer skill registry
([control-skill-registry.md](control-skill-registry.md)) that defines:

- **Skill identity** — stable `skillId` (e.g. `merge-prs`, `launch-batch`)
- **Input schema** — JSON schema per skill
- **Preview phase** — dry-run with no side effects
- **Execute phase** — gated on human confirmation
- **Risk classification** — low/medium/high/critical
- **Audit trail** — every invocation logged
- **Facts produced** — declared knowledge artifacts

This registry governs the WebUI action modules
(`tools/provider-pool-webui/actions/*.js`). Each action module exports
`id`, `label`, `description`, `dangerous`, `preview()`, and `execute()`.

### External Research Intake Loop

External patterns enter LIAN through the research intake loop
([external-research-intake-loop.md](external-research-intake-loop.md)):

1. **Source Capture** — fact event with source class and reliability tier
2. **Evidence Score** — applicability matrix (direct/partial/analogous/aspirational)
3. **Pattern Extract** — LIAN-specific synthesis (not verbatim copy)
4. **Opportunity Signal** — falsifiable hypothesis with bounded experiment
5. **Gate** — agent idea review + human gate
6. **Result Writer** — knowledge entry on experiment completion

This issue (#1495) was generated by `propose-external-intake-issues.js`
converting external research evidence into an investigation proposal.

## Gap Analysis

### What SWE-agent Has That LIAN Lacks

| Dimension | SWE-agent | LIAN Current | Gap |
|-----------|-----------|-------------|-----|
| Tool definition format | YAML per tool | Hardcoded in runner script | No declarative tool definition for self-cycle steps |
| Schema enforcement | Input/output schemas per tool | Scripts validate internally | No cross-tool schema contract |
| Startup discovery | Registry loaded at boot | Runner calls scripts by path | No dynamic tool discovery for self-cycle |
| Adding a new step | Add YAML file | Edit `run-self-cycle.ps1` | Runner modification required |
| Tool versioning | Config version | Script file only | No declarative version tracking |

### What LIAN Already Has

| Dimension | LIAN Capability | Covers SWE-agent Concept? |
|-----------|----------------|--------------------------|
| Skill identity | `control-skill-registry.md` skillId | Yes — governance layer |
| Input schema | WebUI action modules export schemas | Partial — WebUI only, not self-cycle |
| Preview/execute | Every WebUI action has preview + execute | Yes — governance layer |
| Risk classification | low/medium/high/critical per skill | Yes — governance layer |
| Audit trail | `WebUIActionAudit` schema | Yes — governance layer |
| Human gate | Preview → confirm → execute flow | Yes — stronger than SWE-agent |

### The Actual Gap

The gap is **narrow and specific**: the self-cycle runner's five steps
(state-reconciler, health-writer, launch-gate, batch-launch, summary)
are hardcoded as script paths in `run-self-cycle.ps1`. Adding a new step
requires editing the runner. The WebUI action modules already follow a
registry pattern, but the self-cycle runner does not.

## Assessment

### Would a YAML Registry Improve the Self-Cycle Runner?

**Finding: The benefit is marginal and the cost is non-trivial.**

#### Arguments For

1. **Adding a step becomes config-only.** A YAML file per step would let
   operators add pipeline stages without touching the runner script.

2. **Schema visibility.** Declaring input/output schemas per step makes
   the pipeline contract explicit and machine-readable.

3. **Consistency with WebUI actions.** The self-cycle steps would follow
   the same pattern as the WebUI action modules.

#### Arguments Against

1. **Five steps, not fifty.** The self-cycle runner has exactly five
   steps. The YAML registry pattern solves the problem of scaling to
   dozens of tools. At five steps, the hardcoded sequence is readable,
   auditable, and maintainable.

2. **Sequence matters.** The runner's value is the ordered pipeline with
   human decision points between steps. A YAML registry implies
   independent, composable tools — but the self-cycle steps have strict
   dependencies (health state → launch gate → batch launch). The
   orchestration logic would remain in the runner regardless.

3. **Existing registry covers the WebUI surface.** The control skill
   registry already governs the action modules that operators interact
   with. The self-cycle runner is an internal orchestrator, not a
   user-facing tool registry.

4. **Dual truth source risk.** A YAML registry would create a parallel
   declaration that must stay in sync with the scripts. If the YAML says
   a step accepts `healthPath` but the script adds a new parameter, the
   registry drifts. The current approach — script is the source of truth
   — avoids this.

5. **PowerShell/Node.js mix.** The self-cycle steps are a mix of
   PowerShell and Node.js scripts with different invocation patterns
   (some read files, some call GitHub API, some write state). A uniform
   YAML schema would need to accommodate both, adding abstraction
   complexity.

### Verdict

**Do not adopt a YAML tool registry for the self-cycle runner at this
time.** The current architecture is appropriate for the pipeline's
complexity (five steps, strict ordering, human gates). The control skill
registry already provides the governance model for the WebUI surface,
which is the primary operator interface.

## Recommendations

### Recommendation 1: Extend the Control Skill Registry to Self-Cycle Steps

Instead of a new YAML registry, register the self-cycle steps as control
skills in the existing governance model. This gives them:

- Stable identity (e.g. `self-cycle.reconcile`, `self-cycle.health-gate`)
- Input schema declarations
- Risk classification
- Audit trail integration

**Scope:** Documentation update to `control-skill-registry.md` adding
self-cycle step entries to the skill catalogue. No code changes.

**Effort:** Low — the governance model already exists.

### Recommendation 2: Document the Self-Cycle Pipeline Contract

Create a pipeline contract doc that declares:

- Each step's input/output shape
- Step dependencies (which step must precede which)
- Human decision stop points
- Exit codes per step

This gives the schema visibility benefit without a runtime registry.

**Scope:** New doc in `docs/ai-native/`. No code changes.

**Effort:** Low.

### Recommendation 3: Monitor for Registry Need

If the self-cycle runner grows beyond 8–10 steps, or if operators
frequently need to add/remove/reorder steps, revisit the YAML registry
approach. The threshold is when the hardcoded sequence becomes a
maintenance burden rather than a readable pipeline.

**Trigger signals:**

- More than 3 issues requesting new self-cycle steps in a single quarter
- Runner script exceeds 500 lines
- Operators report difficulty understanding step ordering

## Non-Goals

- This investigation does not propose modifying the self-cycle runner.
- This investigation does not propose a new runtime registry.
- This investigation does not propose changes to the WebUI action
  module system.
