# Failure Reflection Step

Reflexion-style self-critique generation for self-cycle failure classification.
Produces actionable lessons from failures so the planning loop can avoid
repeating the same mistakes.

> **Closes:** [#1373](https://github.com/taoyu051818-sys/lian-nest-server/issues/1373)
>
> **Reference:** [classify-self-cycle-failure.js](../../scripts/ai/classify-self-cycle-failure.js)
> for the classifier, [failure-taxonomy-policy.md](failure-taxonomy-policy.md)
> for error class definitions.

---

## Purpose

Current failure classification produces categories (e.g.,
`PROVIDER_UNAVAILABLE`, `WORKTREE_STALE`) and human-readable summaries, but
not actionable lessons. When the same failure recurs, the system has no memory
of what was learned from the previous occurrence.

The reflection step generates a structured self-critique for every
classification, containing:

| Field | Description |
|-------|-------------|
| `lesson` | What went wrong and why — specific to the error class |
| `actionableGuidance` | Concrete steps to prevent this failure next time |
| `repeatPreventionSignal` | Named signal that can be checked before re-attempting |
| `severity` | How impactful repeating this failure would be (`low`/`medium`/`high`) |
| `failureSnippet` | First 200 chars of the failure text for context |
| `capturedAt` | ISO-8601 timestamp |

---

## Usage

### Inline (always present)

Every classification output includes a `reflection` field:

```bash
echo "All providers exhausted" | node scripts/ai/classify-self-cycle-failure.js --step provider-pool-preflight
```

```json
{
  "errorClass": "PROVIDER_UNAVAILABLE",
  "...": "...",
  "reflection": {
    "errorClass": "PROVIDER_UNAVAILABLE",
    "lesson": "All API providers were exhausted, disabled, or at capacity.",
    "actionableGuidance": "Monitor provider pool utilization. Add a pre-launch capacity check.",
    "repeatPreventionSignal": "check-provider-capacity-before-launch",
    "severity": "low",
    "failureSnippet": "All providers exhausted",
    "capturedAt": "2026-05-13T04:30:00.000Z"
  }
}
```

### Persisted to NDJSON log

Pass `--reflectionLog` to append reflections to an NDJSON file for downstream
consumption by the planning loop:

```bash
echo "worktree stale" | node scripts/ai/classify-self-cycle-failure.js \
  --step batch-launch \
  --reflectionLog .github/ai-state/failure-reflections.ndjson
```

Each invocation appends one JSON line. The file grows over time and can be
read by `calculate-meta-signals.js` or the planning loop to detect repeat
failure patterns.

## Integration

```
classify-self-cycle-failure.js   (produces reflection in JSON output)
        |
        ├── stdout                  (reflection included in classification JSON)
        |
        └── --reflectionLog         (appends NDJSON entry for persistence)
                |
                v
        .github/ai-state/failure-reflections.ndjson
                |
                v
        calculate-meta-signals.js   (can consume reflections for scoring)
        plan-next-batch.ps1         (can check repeat-prevention signals)
```

---

## Design Decisions

1. **Always included** — Reflection is not opt-in; every classification
   produces one. This ensures the system always learns from failures.

2. **Deterministic** — Reflections are template-based, not generated by an
   LLM. This keeps the output predictable and testable.

3. **Append-only log** — The NDJSON log is append-only. Each entry is
   self-contained with a timestamp. No deduplication — repeated failures
   produce repeated entries, which is the signal for repeat-failure detection.

4. **Unique signals** — Each error class maps to a unique
   `repeatPreventionSignal`. Downstream consumers can check if a signal was
   recently triggered before re-attempting the same operation.
