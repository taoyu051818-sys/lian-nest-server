# Constitution Amendment Proposal Template

Standard template for proposing changes to the seed constitution. Every
amendment proposal MUST use this format so that reviewers can evaluate
evidence, risks, and rollback plans consistently.

> **Authority:** The seed constitution at
> [`.github/ai-policy/seed-constitution.md`](../../.github/ai-policy/seed-constitution.md)
> is the single source of truth. Amendments also require a mirror update
> to [`seed-constitution.md`](seed-constitution.md).

---

## When To Use This Template

Use this template when:

- A new invariant or boundary must be added to the constitution.
- An existing rule needs modification or clarification.
- A section must be removed because it is obsolete or harmful.

Do **not** use this template for:

- Worker task scoping (use issue CONTROL APPENDIX).
- Policy docs outside the constitution (edit the specific doc).
- Temporary overrides (use the merge gate override flow).

---

## Proposal Sections

Every proposal MUST include all sections below. Incomplete proposals
will be returned to the author.

### 1. Summary

One paragraph describing what the amendment changes and why. Link the
motivating issue or incident.

```markdown
## Summary

Amend <section-name> to <describe change>. Motivated by <issue/incident>.
```

### 2. Proposed Diff

Show the exact text that will change in the constitution file. Use
diff syntax so reviewers can see additions and removals clearly.

```markdown
## Proposed Diff

```diff
- Current rule text.
+ New rule text.
```
```

### 3. Evidence

Provide concrete evidence that the current rule is insufficient or that
the new rule is needed. Evidence classes follow the
[external-reality-intake.md](external-reality-intake.md) reliability
tiers.

```markdown
## Evidence

| # | Source | Class | Reliability | Summary |
|---|--------|-------|-------------|---------|
| 1 | Issue #NNN | github-issue | High | <what happened> |
| 2 | Incident YYYY-MM-DD | human-instruction | Authoritative | <what broke> |
```

### 4. Risk Assessment

Identify risks introduced by the amendment. For each risk, state the
likelihood, impact, and mitigation.

```markdown
## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| <risk description> | Low/Med/High | Low/Med/High | <how to mitigate> |
```

### 5. Red-Team Notes

Anticipate how the amendment could be misused, gamed, or produce
unintended consequences. This section is mandatory even if the answer
is "no foreseeable misuse."

```markdown
## Red-Team Notes

- **Misuse vector:** <how someone could exploit this>
- **Unintended consequence:** <what could go wrong>
- **Mitigation:** <how to prevent or detect>
```

If no misuse vectors are identified, write:

```markdown
## Red-Team Notes

No foreseeable misuse vectors. The amendment <brief reason>.
```

### 6. Simulation

Describe how the amendment was tested or simulated before proposal.
Acceptable forms:

- Dry-run against existing task batches.
- Manual walkthrough of affected worker scenarios.
- Comparison with historical incidents.

```markdown
## Simulation

| Scenario | Method | Result |
|----------|--------|--------|
| <scenario> | dry-run / manual / historical | <outcome> |
```

If no simulation was performed, explain why and note the risk of
proceeding without one.

### 7. Rollout Plan

Describe how the amendment will be activated. Constitution changes take
effect when the PR merges — but downstream guards, scripts, or workers
may need updates.

```markdown
## Rollout Plan

1. Merge constitution PR.
2. Update docs mirror at `docs/ai-native/seed-constitution.md`.
3. <Any guard/script updates needed>
4. <Communication or label changes>
```

### 8. Rollback Plan

Describe how to revert the amendment if it causes problems. Note any
state that cannot be automatically rolled back.

```markdown
## Rollback Plan

1. Revert the constitution PR.
2. Revert the docs mirror PR (if separate).
3. <Any state cleanup needed>
```

---

## Review Requirements

Constitution amendments require all of the following:

| Requirement | Role | Notes |
|-------------|------|-------|
| Author | Human | Amendments MUST be human-authored, not generated by automation |
| Architecture review | `architecture-review` | Validates structural fit and invariant consistency |
| Repository owner approval | `repo-owner` | Final authority on all constitutional changes |
| Docs mirror sync | Author or reviewer | Mirror at `docs/ai-native/seed-constitution.md` must be updated in the same PR or a linked PR |

---

## Submission Checklist

Before submitting an amendment proposal:

- [ ] All 8 proposal sections are filled in.
- [ ] Evidence cites at least one verifiable source.
- [ ] Risk assessment covers likelihood and impact.
- [ ] Red-team notes section is present (even if "no vectors").
- [ ] Simulation section describes testing or justifies absence.
- [ ] Rollout and rollback plans are actionable.
- [ ] Proposal is human-authored (not generated by automation).
- [ ] Docs mirror update is included or linked.
